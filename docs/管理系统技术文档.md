# OmniSpark AI 管理系统 — 完整技术文档

## 1. 项目概述

OmniSpark AI 是一个统一生图与视频创作管理平台。用户在一个账号内完成文生图、图生视频、模型配置、素材管理、团队协作等全流程创作。

### 1.1 技术栈

| 层级 | 技术 | 版本 |
|------|------|------|
| 前端框架 | Vue 3 (Composition API + `<script setup>`) | ^3.5 |
| 构建工具 | Vite | ^8.0 |
| UI 组件库 | Naive UI | ^2.44 |
| 状态管理 | Pinia | ^3.0 |
| HTTP 客户端 | Axios | ^1.16 |
| 图标库 | Lucide Vue Next | ^1.0 |
| 后端框架 | Spring Boot | 3.0.5 |
| ORM | MyBatis-Plus | 3.5.15 |
| 数据库 | MySQL | 8.0+ |
| 缓存/会话 | Redis | 7.x |
| 鉴权 | Sa-Token | 1.45.0 |
| 工具库 | Hutool | 5.8.36 |
| JVM | Java | 17+ |

### 2.3 状态管理 (Pinia Store)

**userStore** — 用户身份
```typescript
state: { isLoggedIn, userInfo, token }
actions: { setSession(), logout() }
```

**projectStore** — 项目空间
```typescript
state: { projects[], activeProjectId }
actions: { refresh(), addProject(), deleteProject() }
```

**taskStore** — 任务队列
```typescript
state: { tasks[] }

```

**providerStore** — 模型提供商
```typescript

```

**assetStore** — 资产库
```typescript

```

### 2.4 API 请求封装 (request.ts)

Axios 实例配置：
- `baseURL`: 环境变量 `VITE_API_BASE_URL` 或 `http://localhost:8080`
- `timeout`: 15 秒
- 请求拦截器：自动附加 `satoken` 到请求头
- 对 `/api/auth/**` 与管理员用户写接口自动追加 `X-Timestamp`、`X-Nonce`、`X-Sign`
- JSON 请求按稳定 key 顺序签名；`application/x-www-form-urlencoded` 请求按排序后的表单参数串签名，避免框架包装导致的验签偏差
- 响应拦截器：统一错误处理，成功响应自动解包
- **SWR 缓存**：GET 请求 15 秒内存缓存，`clearCache()` 可手动清除

### 2.5 组件通信规范

- 父子组件：Props + Emits
- 全局状态：Pinia Store
- 页面间传参：Vue Router query 参数（如生图页接收 prompt/model/size 参数）

### 3.2 统一响应格式

```json
{
  "code": 200,
  "message": "操作成功",
  "data": { ... }
}
```

成功：`code=200`，`data` 为业务数据
失败：`code=500`，`message` 为错误描述

### 3.3 异常处理

| 异常 | 说明 | HTTP 状态码 |
|------|------|-------------|
| `BusinessException` | 业务逻辑错误（如账号不存在） | 500 |
| `MethodArgumentNotValidException` | 参数校验失败 | 500 |
| `ConstraintViolationException` | 约束违规 | 500 |
| `Exception` | 其他未预期异常 | 500 |

### 3.4 鉴权体系

- **登录**：`StpUtil.login(userId)` → 返回 token
- **前端存储**：`localStorage.setItem('satoken', token)`
- **请求传递**：Axios 拦截器自动附加 `satoken` 请求头
- **敏感接口防篡改**：认证接口与管理员用户写接口额外启用 HMAC 签名 + Redis nonce 防重放
- **后端验证**：`@SaCheckLogin` 注解保护接口
- **角色**：`StpInterface` 从数据库 `user.role` 字段加载角色
- **项目级权限**：`@RequireProjectPermission("view"|"edit"|"admin")` 注解

### 3.5 权限层级

```
admin  >  edit  >  view
  3        2         1
```

- `view`：查看项目内容
- `edit`：编辑项目内容
- `admin`：管理项目共享设置

### 3.6 数据可见范围

- 普通业务数据默认按**项目隔离**，用户仅可查看自己拥有的项目数据。
- 若项目通过 `project_share` 共享给某个团队，则该团队成员可按共享权限查看或编辑该项目数据。
- **公共内容例外**：提示词模板、角色/风格卡、社区共享帖子采用全员可发布、全员可见策略，不受项目可访问范围限制。
- 公共内容会展示**发布者身份**，并支持**点赞、评论、回复**等互动能力。
- 资产、模型提供商、工作流、任务、字幕等仍统一按“本人项目 + 团队共享项目”过滤。
- 生成链路中的 `providerId`、参考图、遮罩图、视频源图、尾帧图等关联资源，也要求与当前项目归属一致，不能跨项目混用。

## 6. 功能模块详解

### 6.1 生图流程

```
用户填写表单 → 点击生成
  → generationApi.generateImage(dto)
  → POST /api/generation/image
  → GenerationService.createTask() 创建任务
  → OpenAiImageClient 调用 OpenAI 兼容 API
  → 保存结果到 asset 表
  → 更新任务状态为 success
  → 推送通知到 WebSocket
```

### 6.2 局部重绘流程

```
用户上传参考图 → 在 Canvas 上绘制遮罩（白色区域=重绘区域）
  → 遮罩图以 PNG 上传到资产库
  → POST /api/generation/image/inpaint (带 maskAssetId)
  → 后端加载原图 + 遮罩图
  → 调用 OpenAI /images/edits API（带 mask 参数）
  → 返回重绘结果
```

### 6.3 团队协作流程

```
创建团队 → 邀请成员（通过用户名）
  → 被邀请用户收到通知
  → 团队所有者可共享项目给团队
  → 项目共享设置权限（view / edit / admin）
  → 团队成员自动获得对应权限
  → 在项目列表中看到共享项目
  → 共享项目下的资产、模型、工作流、任务等项目数据对团队成员可见
```

### 6.4 字幕/配音流程

```
视频资产详情 → 「从提示词生成字幕」
  → 后端将 prompt 按句子分割 + 分配时间轴 → 生成 SRT
  → 可编辑 SRT 内容
  → 「配音」→ 保存文本 → 返回音频 URL（可接入 TTS API）
  → 在资产详情中播放配音
```

### 6.5 顺序工作流

```
用户定义 JSON 步骤数组：
  [{"type":"image","prompt":"..."}, {"type":"video","prompt":"..."}]
→ 保存为 Workflow
→ 点击「执行」→ 后端依次执行每步
→ 记录运行结果到 workflow_run
→ 查看运行历史
```

### 6.6 公共内容互动

```
登录用户发布提示词模板 / 风格卡 / 社区帖子
  → 内容记录作者信息（userId / username / nickname / avatar）
  → 其他登录用户可浏览、点赞、评论、回复
  → 回复输入框默认预填 @昵称/用户名
  → 评论作者可删除自己的评论；删除根评论时会一并删除其回复
  → 点赞数 / 评论数回写到公共内容卡片
  → 模板、风格卡、社区帖子支持按最新、点赞数、评论数排序
  → 模板、风格卡、社区帖子仅作者本人可继续编辑或删除
```

- 提示词模板库与风格卡库对外都属于“公共资源库”。
- 当前后端内部仍复用保留槽位 `project_id = 0` 做存储隔离，但这只是实现细节，已经统一收敛到公共库约定，不再作为接口语义暴露。
- 模板标签与风格卡标签接口返回的是当前公共内容中的实际标签集合，用于筛选与输入建议，不属于后台固定字典。
- 社区分类由 `community_category` 数据字典提供，发布页只允许选择启用中的分类项，避免“没有帖子就没有分类”或由历史帖子反推分类选项。
- 对外字典查询当前统一走 `/api/dict/{dictCode}/items`，只返回启用项；旧的“返回全部项”公开路径已下线，避免同类接口出现两套语义。

### 6.7 系统初始化边界

- `BootstrapDataInitializer` 仅补齐系统运行所需的基础记录，例如管理员账号、平台配置、资产分类字典与默认清理任务。
- 启动过程不会写入演示项目、演示任务、演示素材或其他业务样例数据，因此这部分不应视为“假数据”。

### 6.8 定时任务边界

- 当前调度器不是通用任务市场，只开放了真实已实现的 `cleanup` 数据清理任务。
- 前端创建页、后端校验与运行逻辑都只接受 `cleanup`，因此这里属于能力范围有限，而不是“假接口”。

### 6.9 管理端任务筛选边界

- 管理端任务状态与任务类型筛选项属于固定系统枚举，目前稳定限定为 `pending/running/success/failed` 与 `image/video`。
- 该筛选元数据不再从历史任务表反推，避免“没有历史数据就没有筛选项”的半动态行为。

### 6.10 审计动作筛选边界

- 审计动作筛选项来自当前代码可记录的控制器写操作目录，而不是来自 `audit_log` 历史记录去重。
- 这保证了动作筛选不会因为日志尚未产生、用户从未触发、或历史日志被清理而消失。
- 审计切面与动作目录共用同一套动作命名 / 判定规则，并显式排除 GET 只读接口，避免“接口列出的动作”和“实际写入日志的动作”逐渐漂移。

### 6.11 系统更新能力边界

- 管理端“系统更新”依赖 GitHub 仓库 owner/repo 配置，或本地 `.git origin` 元数据推断出的仓库信息。
- 若更新源未配置，接口会明确返回“当前不可用”的能力状态；此时页面只展示本地构建信息，不再把更新检查伪装成可用能力。
- 当前更新来源优先级为 Release，其次 Tag，再其次默认分支最新提交；因此它是“真实能力但依赖外部配置与仓库状态”的后台工具，不属于假接口。

### 6.12 系统日志能力边界

- 管理端“系统日志”读取的是后端日志目录中的真实 `.log` 文件，不是演示日志或固定样例。
- 若日志文件名非法、日志目录不存在、或当前无可读日志文件，接口会显式返回“不可用”状态，而不是把错误说明伪装成日志正文。

### 6.13 文件管理能力边界

- 管理端“文件管理”浏览的是后端上传目录中的真实落盘文件，不是演示文件列表或数据库拼装占位数据。
- 若上传目录尚未初始化、上传目录配置无效、或请求的子目录不存在，接口会显式返回目录状态说明，而不是统一伪装成接口异常。

### 6.14 性能监控能力边界

- 管理端“性能监控”展示的是 JVM 进程与宿主机资源的轮询快照，适合快速排查当前运行状态。
- 当前实现不提供历史趋势、网络探针、容器编排指标或告警编排，因此它属于轻量监控页，而不是完整观测平台。

### 6.15 系统健康检查边界

- 管理端“系统状态”当前只检查数据库、Redis、构建版本与 JVM 运行时，不覆盖对象存储、第三方模型 API、Webhook 下游、邮件短信或完整业务链路巡检。
- 因此 `/api/admin/health` 属于基础运行时自检接口，而不是完整健康中心或全依赖巡检平台。

### 6.16 Webhook 能力边界

- 管理端 Webhook 当前只支持固定三类任务事件：`task.started`、`task.completed`、`task.failed`。
- `/api/admin/webhooks/meta` 返回的是代码内固定事件集，不会随着数据库记录或用户配置自动扩展，因此它不是动态订阅中心。

### 6.17 数据清理能力边界

- 管理端“数据清理”当前只处理旧任务、旧资产、审计日志、登录日志，并同步删除关联资产文件及部分引用关系。
- 它不负责清理用户、项目、系统配置、公共内容或外部对象存储，因此属于范围有限的后台治理工具，而不是全库归档中心。

### 6.18 管理仪表盘能力边界

- 管理端“管理控制台”展示的是当前库内概览、最近 7 天趋势和最近注册用户快照，用于快速运营查看。
- CSV 导出也只覆盖这组当前快照，不提供完整历史归档、分项目钻取分析或自定义报表编排，因此它不是完整 BI 报表中心。

### 6.19 资产统计能力边界

- `/api/assets/stats` 当前只返回访问范围内的数量快照：总资产数、图片数、视频数、参考素材数、收藏数。
- 它不提供存储占用、体积分布、历史趋势、下载热度或生命周期分析，因此不应被视为完整资产分析中心。

### 6.20 管理端资产监管边界

- 管理端“全局资产监管”当前聚焦全局列表浏览、文件预览和删除。
- 它不承担完整的资产成本分析、生命周期治理、冷热分层或趋势报表职责，因此更接近后台监管页，而不是全功能资产运营中心。

### 6.21 用户侧统计能力边界

- `/api/stats/dashboard` 当前只返回当前访问范围内的概览、最近 14 天趋势、项目活跃排行和最近 8 条活动。
- 它不提供完整审计流水、逐小时计费明细、任意时间范围切片或长期历史归档，因此属于用户侧用量快照，而不是完整分析中心。

### 6.22 通知中心边界

- 当前通知能力以 WebSocket 推送为主，并由前端 30 秒轮询做补偿刷新，保证在实时连接不可用时仍能回收大部分通知状态。
- 顶栏通知弹窗只保留最近一小段列表数据，用于快速查看与跳转，不承担完整历史通知归档中心职责。

### 6.23 工作流能力边界

- `/api/workflows/meta` 返回的是代码内固定三类步骤配置：`image`、`video`、`subtitle`，以及默认尺寸、时长、字幕语言。
- 工作流执行器当前按步骤数组顺序逐步执行，并可继承上一步产物作为下游输入。
- 运行记录当前只保存执行状态、当前步骤、步骤结果和错误信息，用于查看线性执行结果。
- 当前不支持分支、并行、条件判断、重试编排、人工审批、外部插件节点或通用 DAG 调度，因此它属于轻量顺序执行器，而不是完整工作流平台。
