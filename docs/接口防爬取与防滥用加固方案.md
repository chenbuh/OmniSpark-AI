# 接口防爬取与防滥用加固方案

> 目标：防止接口被批量爬取数据（社区帖、公共模板、风格卡库）、被脚本盗刷拖垮服务端资源、被撞库爆破。
> 本文同时作为**认知澄清 + 现状盘点 + 落地方案**。立论基于对本项目 `backend` 源码的实际审查（2026-05-31）。
>
> **重要前提（决定了威胁模型）**：本项目的 AI 供应商 API Key 由**用户自行配置**，调用生成接口消耗的是**用户自己的第三方额度，不是平台成本**。因此本文不把"AI 费用被盗刷"当作风险——生成接口仍需限频，但理由是**保护平台自身的服务器资源**（CPU、数据库连接、任务线程池、上传磁盘、出网带宽），而非省钱。
>
> **落地状态（2026-06）**：首轮防爬加固已实现 —— 读取接口默认 `IP + USER_API` 双维度限流、可信代理真实 IP 解析、API Key scope/过期/额度/IP-UA/冻结风控、服务端短期签名 challenge、访问日志聚合、异常访问临时降速/二次验证码、导出 canary 水印均已接入。详见 `docs/防爬加固TODO.md`。

## 0. 先破除一个误区：浏览器里「藏接口」做不到

很多人第一反应是"让别人在 F12 里看不到接口"。**这个目标在技术上不可能实现**，必须先讲清楚，否则方向就错了：

- 浏览器要调用接口，请求就必须真实发出去。F12 的 Network 面板抓的是**浏览器自己发出的真实流量**，只要前端能调通，F12 就一定看得到完整的 URL、请求头、参数和响应体。这是 HTTP 的工作原理，不是能力问题。
- 常见的"伪隐藏"手段——参数加密、响应加密、JS 代码混淆、禁用右键/禁用 F12、检测开发者工具——**只能提高逆向的时间成本，挡不住有心人**。它们对"路过的脚本小子"有点用，对"愿意花十分钟读 JS 的人"无效。
- 本项目曾经也存在这个问题：前端包里放过长期签名凭据。现在已改为服务端短期 challenge，前端每次敏感写请求只拿一次性 challenge 凭据，后端消费后验签，避免长期静态凭据被逆向后复用。

**正确的思路不是"藏接口"，而是"接口暴露了也让滥用不划算"**：
> 假设攻击者完全知道你所有接口的地址和参数（事实上他确实能知道），你的系统是否还安全、还扛得住批量请求？

这套防护的核心是五层：**认证 → 限流 → 配额 → 数据最小化 → 审计追溯**。下面先盘点本项目现状，再逐层给方案。

| --- | --- | --- | --- |
| **P0** | **全项目无任何限流** | grep `RateLimit/Bucket/限流` 无任何匹配 | 脚本可无限狂刷任意接口：撞库爆破登录、批量爬取数据、并发塞满任务线程池与数据库连接拖垮服务 |
| **P0** | **生成接口不做频率/并发前置校验** | `GenerationService` 只在生成**后** `quotaRecordMapper.insert` 记流水，调用前从不检查频率或在跑任务数 | 登录用户可瞬间发起海量生成任务，占满后端任务线程池、上传磁盘与出网带宽，正常用户被拖到不可用（**注意：消耗的 AI 额度是用户自配 Key 的，不是平台成本；此处护的是平台服务器资源**） |

| **P2** | **无安全响应头** | grep `X-Frame-Options/CSP/X-Content-Type-Options` 无匹配 | 易被点击劫持、MIME 嗅探、嵌入 iframe 钓鱼 |
| **P2** | **公开接口（社区）可被全量爬取** | 社区列表匿名/登录可访问且无限流 | Prompt + 效果图被批量采集 |

> 一句话总结现状：**门锁（认证）装得不错，但没有限流这道"防洪闸"，一旦有人写脚本，登录接口会被撞库、列表接口会被拖库、生成接口会被并发塞爆任务队列拖垮整个服务。**

## 2. P0 方案一：接口限流（最关键，优先做）

限流是防爬取/防盗刷的第一道闸，也是当前最大的缺口。原理：基于 Redis 对「单位时间内某主体的请求次数」计数，超过阈值就拒绝。本项目已有 Redis，零新增中间件。

### 2.1 限流维度（按主体分别限）

| 维度 | 适用接口 | 建议阈值（起步值，按实际调） |
| --- | --- | --- |
| **按 IP** | 登录、注册、公钥等匿名接口 | 登录 10 次/分钟/IP；注册 5 次/小时/IP |
| **按用户** | 所有需登录接口 | 通用 120 次/分钟/用户 |
| **按用户 + 接口** | 生成类高成本接口 | 生图/生视频 10 次/分钟/用户、200 次/天/用户 |
| **全局** | 兜底总闸 | 单实例 QPS 上限，防异常洪峰 |

### 2.2 推荐实现：注解 + 拦截器（Redis 滑动窗口/固定窗口）

最小侵入做法：自定义 `@RateLimit` 注解 + 一个 `RateLimitInterceptor`，用 Redis `INCR + EXPIRE` 做固定窗口计数。示意：

```java
// 注解
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface RateLimit {
    int count();                 // 窗口内允许的次数
    int seconds();               // 窗口大小（秒）
    Dimension dimension() default Dimension.USER;  // IP / USER / USER_API
    String message() default "操作过于频繁，请稍后再试";
}

// 拦截器核心逻辑（伪代码）
String key = "rate:" + dimension + ":" + subject + ":" + handlerKey;
Long current = redis.opsForValue().increment(key);
if (current == 1L) redis.expire(key, Duration.ofSeconds(seconds));
if (current > count) {
    response.setStatus(429);  // Too Many Requests
    write(response, message);
    return false;
}
return true;
```

> 固定窗口实现简单、足够用；若担心窗口边界突刺，可升级为 Redis + Lua 的滑动窗口或令牌桶（如引入 `bucket4j-redis`）。起步阶段固定窗口即可。

### 2.3 落地步骤

1. 新增 `common/annotation/RateLimit.java` 与 `common/config/RateLimitInterceptor.java`。
2. 在 `WebConfig.addInterceptors` 注册（顺序放在 `ApiSignInterceptor` 之后、业务之前）。
3. 在关键接口打注解：
   - `AuthController.login` → `@RateLimit(count=10, seconds=60, dimension=IP)`
   - `AuthController.register` → `@RateLimit(count=5, seconds=3600, dimension=IP)`
   - `GenerationController` 各生成方法 → `@RateLimit(count=10, seconds=60, dimension=USER_API)`
4. 前端对 `429` 统一提示「操作过于频繁」（`request.ts` 响应拦截器加一个分支即可）。
5. 阈值先松后紧：上线初期设宽松值观察 `login_log` 与审计日志，再逐步收紧。

### 2.4 取主体 IP 的注意点

生产若在 Nginx/网关后，`request.getRemoteAddr()` 拿到的是网关 IP。需读 `X-Forwarded-For` 第一段，并**只信任自己网关注入的值**（否则攻击者可伪造该头绕过 IP 限流）。在网关层强制覆写 `X-Forwarded-For` 是标准做法。

### 2.5 落地状态（已实现）

限流已按本节方案落地，对应代码：

- `common/annotation/RateLimit.java` —— 限流注解。字段：`count`（窗口内次数）、`seconds`（窗口秒数）、`dimension`（`IP` / `USER` / `USER_API`）、`message`（超限文案）。
- `common/annotation/RateLimits.java` —— 容器注解，支持在同一方法上叠加多条 `@RateLimit`（如「10 次/分钟」+「200 次/天」双闸）。
- `common/config/RateLimitInterceptor.java` —— 拦截器，Redis `INCR + EXPIRE` 固定窗口计数；超限返回 `429` 与 JSON 文案；`USER_API` 维度的主体是「用户 + 请求路径」。已在 `WebConfig` 注册，顺序在 `ApiSignInterceptor` 之后。
- IP 主体优先取 `X-Forwarded-For` 第一段，回退 `getRemoteAddr()`（生产需在网关强制覆写该头，见 §2.4）。

已挂注解的接口：

| 接口 | 限流 |
| --- | --- |
| `POST /api/auth/login` | 10 次/分钟/IP |
| `POST /api/auth/register` | 5 次/小时/IP |
| `GET/POST /api/auth/captcha/**` | 30 次/分钟/IP |
| `POST /api/generation/image`、`/video`、`/image/inpaint` | 10 次/分钟/用户 + 200 次/天/用户（双闸叠加） |

> 前端 `request.ts` 响应拦截器已能透传后端文案，`429` 会以 `message` 形式提示用户「操作过于频繁」。

## 3. P0 方案二：生成接口的并发与频率前置控制（护服务器，非省钱）

**先明确边界**：本项目 AI 供应商 Key 由用户自配，生成调用烧的是用户自己的第三方额度，平台不为此付费。所以这里的目标**不是省 AI 钱，而是防止单个用户瞬间发起海量生成任务，把平台自己的资源占满**——这些才是平台的真实成本：

- **后端任务线程池**：`application.yml` 中 `task.scheduling.pool.size: 5`，并发生成任务有限，被一个人塞满后所有人排队。
- **数据库连接 / 行写入**：每个任务都要写 `generation_task`、轮询更新进度、写 `asset`。
- **上传磁盘与出网带宽**：参考图下载、结果回源下载、缩略图生成全部走平台服务器。

当前 `GenerationService` 只在生成完成后 `quotaRecordMapper.insert` 记一笔流水，**生成之前从不做任何频率或并发校验**。配额只是"账单"，不是"闸门"。

### 3.1 改造点（两道闸，按需取用）

**闸门 A — 频率限流（复用第 2 节的 `@RateLimit`）**：给 `generateImage / generateVideo / inpaint` 打 `USER_API` 维度注解，例如 10 次/分钟/用户，挡住"瞬间狂发"。这是最简单、收益最大的一步，做了第 2 节就免费获得。

**闸门 B — 在跑任务数上限（防线程池被单人占满）**：限制单用户"同时处于排队/执行中的任务数"，而不是总量。

> 注意：当前 `generation_task` 表**不带 `user_id` 字段**（userId 仅作为参数传入 `submitGenerationJob`，并写入 `quota_record`）。任务状态实际取值为 `running / success / failed`。因此"按用户统计在跑任务数"**不能直接查 `generation_task` 表**，推荐用 Redis 原子计数实现：

```java
public GenerationTaskVO generateImage(ImageGenerateDTO dto) {
    projectAccessGuard.assertAccess(dto.getProjectId());
    Long userId = SecurityUtil.loginUserId();
    String key = "gen:running:" + userId;
    Long running = redis.opsForValue().increment(key);          // 提交即 +1
    redis.expire(key, Duration.ofMinutes(15));                  // 兜底防泄漏（任务最长 10 分钟）
    if (running != null && running > MAX_CONCURRENT_PER_USER) {  // 如 3
        redis.opsForValue().decrement(key);                     // 超限回滚
        throw new BusinessException("您有较多生成任务正在进行，请等待完成后再提交");
    }
    // ... 提交任务；在 submitGenerationJob 的 finally / 任务结束回调里 redis.decrement(key)
}
```

> 关键是任务无论成功/失败/超时都要在结束时 `DECR`，否则计数只增不减会把用户永久锁死——所以放在 `submitGenerationJob` 的 `finally` 块里递减最稳妥。若要更省事，也可以只做闸门 A（频率限流），它已能挡住绝大多数"瞬间狂发"。

### 3.2 配套

- **阈值可配置**：`MAX_CONCURRENT_PER_USER`、每分钟次数等提到 `system_config` 表或 `application.yml`，便于按负载调整。
- **并发安全**：高并发下"查-判-写"有竞态。要严格的话用 Redis 原子 `INCR` 维护"在跑计数"，任务结束时 `DECR`，避免绕过。
- **业务限额（可选，与本前提无关）**：如果未来有产品层面的"免费用户每天 N 张"诉求，再单独做总量配额——那是商业策略，不是安全防护，两者不要混为一谈。
- 限流（第 2 节）防"短时间高频"，在跑任务上限防"单人占满线程池"，两者互补。

## 4. P1 方案：分页上限 + 登录失败锁定

### 4.1 分页大小强制上限（防全量爬取）

所有列表接口的 `pageSize` 当前无上限，可被 `pageSize=99999999` 一次拖库。统一加一个工具方法夹紧：

```java
public static long clampPageSize(long pageSize) {
    if (pageSize < 1) return 10;
    return Math.min(pageSize, 100);   // 上限 100，按需调整
}
```

在每个 `selectPage(new Page<>(page, clampPageSize(pageSize)), ...)` 处套用。涉及：`UserAdminController`、`AdminTasksController`、`AdminAssetsController`、`AdminStatsController`、社区/模板/风格卡列表等。**这是改动小、收益大的一步。**

### 4.2 登录失败锁定（防爆破）

在 `AuthService.login` 失败分支加 Redis 计数：

```
key = login:fail:{username}（或 {ip}）
每次密码错误 INCR；连续失败 ≥ N 次（如 5 次）→ 锁定 M 分钟（如 15 分钟），期间直接拒绝
登录成功 → DEL 该 key
```

进阶：失败 3 次后要求图形验证码（前端集成滑块/字符验证码），平衡安全与体验。配合第 2 节的登录 IP 限流，撞库基本被堵死。

### 4.3 自研多形态验证码（登录 / 注册前置，已实现）

在 §4.2 的失败锁定之外，登录与注册都前置了一道**自研滑块验证码**。不用第三方、不用任何图片素材，图料全部由后端 `BufferedImage` 纯代码生成。

#### 4.3.1 为什么自研 + 多形态

诉求是「与众不同、让脚本难破」。先讲清楚边界：**纯前端验证码没有"不可破解"，只有"破解成本高低"**——代码和图都在浏览器里，F12 全看得到。真正决定难度的不是图案多奇特，而是这三点，本实现都做了：

1. **答案只在后端**：旋转的正确角度、点击目标的有序坐标、轨道终点 x，全部只写入 Redis，**绝不下发给前端**。脚本只能靠图像识别去"猜"，而不是读一个前端变量。
2. **多形态随机**：每次请求随机出一种形态，攻击者无法只针对一种写解法通吃。
3. **行为风控**：校验"像人"的作答轨迹 + 按 IP 记失败次数动态拒发。

#### 4.3.2 三种形态（后端每次随机一种）

| 形态 | type | 玩法 | 后端答案 | 通过判定 |
| --- | --- | --- | --- | --- |
| 旋转校正 | `rotate` | 圆盘被随机转了 N 度，拖滑条转正 | 正确角度 | 作答角度与正确角度差 ≤ ±12° |
| 顺序点击 | `sequence` | 按提示顺序点击图上 3 个形状（方块/三角/圆/星/菱形） | 各目标中心的有序坐标 | 每次点击落在对应目标中心 ≤ 28px，且顺序一致 |
| 轨迹拖动 | `track` | 沿后端随机生成的折线轨道把滑块拖到终点 | 终点 x | 落点与终点 x 差 ≤ ±8px |

#### 4.3.3 完整流程

```
1. 前端点登录/注册 → 先校验表单 → 弹出验证码组件
2. GET  /api/auth/captcha/generate
     后端随机选形态，画图，把【答案】写入 Redis(captcha:answer:{id}, TTL 2 分钟)
     只返回 captchaId + type + 图料 + 提示语（不含答案）
3. 用户作答，组件全程采集行为轨迹 [x, y, 相对时间]
4. POST /api/auth/captcha/verify {captchaId, 作答结果, trail}
     后端先删答案（一次性，无论成败）→ 行为风控 looksHuman → 按形态判定
     通过 → 发一次性票据 captchaTicket(captcha:ticket:{t}, TTL 2 分钟)
5. 前端拿票据，连同账号密码提交 /api/auth/login(或 register)
6. AuthController 入口 consumeTicket：票据存在且未用过才放行，否则「请先完成滑块验证」
```

要点：
- **答案一次性**：`verify` 一进来就删 `captcha:answer:{id}`，杜绝同一张图重复试错爆破。
- **票据一次性**：登录/注册时 `consumeTicket` 用 Redis `delete` 的返回值判定，删成功才算有效，天然防重放。
- **验证与登录强绑定**：没有有效票据，光有正确账号密码也进不来。

#### 4.3.4 行为风控（`looksHuman`，按形态分级）

不是所有形态都要求长轨迹（点击/旋转可能"看一会儿再快速作答"），所以按形态设最小采样点与规则：

- **通用**：相对时间必须单调不减；从验证码展示到作答的总耗时 ≥ `MIN_HUMAN_MILLIS`（防瞬时脚本）。
- **最小轨迹点**：rotate 1 点、sequence 等于目标数、track ≥ 4 点。
- **track 额外**：横向位移幅度要够（`maxX-minX ≥ 80`，防原地假拖）；时间间隔与位移步长不能过于均匀（脚本常等步长，真人有抖动与加减速）。
- **IP 失败计数**：同 IP 在 10 分钟内累计失败 ≥ 20 次，临时拒发新验证码（`captcha:fail:{ip}`）。

> 阈值偏宽松是**有意为之**——起步优先不误伤真人。要更严可加：鼠标加速度分布、贝塞尔轨迹拟合度、设备指纹。这些在 `CaptchaService` 里集中可调。

#### 4.3.5 代码位置

## 5. P2 方案：安全响应头 + 公开接口治理

### 5.1 补齐安全响应头

全项目当前无任何安全响应头。加一个 `SecurityHeadersFilter`（或在 Nginx 层统一注入）：

| 响应头 | 值 | 作用 |
| --- | --- | --- |
| `X-Content-Type-Options` | `nosniff` | 禁止 MIME 嗅探 |
| `X-Frame-Options` | `DENY` | 防点击劫持（禁止被 iframe 嵌入） |
| `Content-Security-Policy` | 按需配置 | 限制脚本/资源来源，防 XSS 注入外链 |
| `Referrer-Policy` | `no-referrer-when-downgrade` | 控制 referer 泄露 |
| `Strict-Transport-Security` | `max-age=31536000` | 强制 HTTPS（仅在 HTTPS 下发） |

### 5.2 公开接口（社区）的特殊处理

社区列表是对外可见的，最容易被批量采集 Prompt + 效果图。除限流外：

- **图片防盗链**：`uploads/**` 静态资源校验 `Referer`，非本站来源拒绝（Nginx 一行配置）。
- **列表字段最小化**：列表接口只返回卡片必需字段，详情才返回完整内容，抬高"一次拖全量"的成本。
- **可选**：对未登录用户降低可见范围或要求登录后才能看完整 Prompt。

### 5.3 传输层地基：强制 HTTPS

没有 HTTPS，前面所有努力都可被中间人架空（替换公钥、嗅探 token）。生产必须：Nginx 配 TLS + HSTS，HTTP 301 跳 HTTPS。这是地基，不是可选项。

## 7. 关于「加密接口 / 隐藏接口」的最终结论

回到最初的诉求，诚实地总结：

- **「让别人在浏览器看不到接口」做不到**，这是 HTTP 的本质决定的。任何宣称能做到的方案都是把"提高逆向成本"包装成了"隐藏"。
- **能做且值得做的是"防滥用"**：假设接口完全公开，靠认证 + 限流 + 并发控制 + 数据最小化 + 审计，让批量爬取和盗刷在技术上被拦截、在资源上占不到便宜。
- **本项目最该补的是限流和生成接口并发控制**——门锁已经不错，但缺了防洪闸。补上这两块，"别人写脚本爬数据/塞爆服务"的问题就解决了大半。
- **关于成本的澄清**：AI 供应商 Key 是用户自配的，盗刷烧的是用户自己的额度，不是平台的钱。所以平台要护的不是"AI 账单"，而是**自己的服务器资源**（线程池、数据库、磁盘、带宽）和**数据资产**（社区/模板/风格卡内容不被批量爬走）。
- 没有绝对安全。安全工程的目标始终是**让攻击成本远高于收益**，而不是追求一个不存在的"绝对隐藏"。
