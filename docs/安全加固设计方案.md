# 安全加固设计方案

> 本文档为**设计/实施方案**，不含落地代码改动。涵盖三块：密码存储哈希加固、登录接口防护（传输加密 + 请求签名 + 防重放）、自研滑块缺口验证码。
>
> 适配现状：后端 Spring Boot 3 + Sa-Token + MyBatis-Plus + Redis + Hutool；密码已用 BCrypt（`PasswordUtil`，兼容历史无盐 SHA-256 并在登录成功后平滑升级）；前端 Vue 3 + naive-ui，登录/注册接口为 `/api/auth/login`、`/api/auth/register`。

---

## 0. 先厘清一个常见误区

「让用户 F12 看不到接口和密码」与「数据库里的密码哈希强度」是**两个不同层面**的问题，常被混为一谈：

| 关注点 | 能做到什么 | 做不到什么 |
| --- | --- | --- |
| 传输层加密 / 请求签名 | 让 F12 Network 里抓到的是**密文**，明文密码不落在请求体；篡改和重放被拒 | 无法让前端 JS 代码本身「看不见」——浏览器必须能执行它，源码总能被读到 |
| 密码存储哈希 | 即使**拖库**，攻击者也难以从哈希反推明文 | 与前端能否看到接口无关 |

结论先行：

1. **前端代码必然可被 F12 读到**，任何「加密逻辑」都在客户端可见。能做的是提高门槛（传输加密、签名、防重放），不是根除。
2. 用户说的「5090 显卡 1 小时就能算出来的加密」指的是**弱哈希被 GPU 高速暴力破解**——典型是无盐 SHA-256 / MD5（每秒可算百亿次），而非加盐慢哈希。本项目当前用 BCrypt 已能抵御，方案是**进一步提高 cost 因子**，下文详述。

---

## 1. 密码存储哈希加固（BCrypt 提高 cost）

### 1.1 为什么 SHA-256 / MD5 顶不住 GPU

- MD5、SHA-256 是**为快而设计**的通用摘要算法。单张 RTX 5090 量级的显卡，针对无盐 SHA-256 每秒可尝试 **百亿（10^10）量级** 的口令。
- 8 位常见口令（字母+数字，约 62^8 ≈ 2.2×10^14 组合）在这种速度下，配合字典和规则，**小时级即可大面积命中**。
- 无盐还会被**彩虹表**直接查表，连暴力都省了。

本项目历史上确实存在过「无盐 SHA-256，且 64 位 hex 原样当哈希」的问题（见 `docs/代码审查与安全审计报告.md`），现已改为 BCrypt 并平滑升级。

### 1.2 BCrypt 为什么能抗 GPU

- BCrypt 基于 Blowfish 的密钥扩展，**故意慢**且**内存访问不规则**，对 GPU 大规模并行不友好。
- 自带 **128 bit 随机盐**，彩虹表失效，每个用户哈希唯一。
- **cost 因子**（也叫 work factor）以 2 的幂控制迭代轮数：cost 每 +1，计算耗时翻倍。

> Hutool 的 `BCrypt.hashpw(rawPassword)` 默认 cost = 10（即 2^10 = 1024 轮）。要提高强度，需显式指定盐：`BCrypt.hashpw(raw, BCrypt.gensalt(12))`。

### 1.3 cost 因子怎么选

| cost | 相对耗时 | 单次哈希参考耗时* | 说明 |
| --- | --- | --- | --- |
| 10 | 1x | ~50–100ms | Hutool 默认，偏低 |
| **12** | 4x | ~200–400ms | **推荐值**，安全与体验平衡 |
| 13 | 8x | ~400–800ms | 高安全场景 |
| 14 | 16x | ~1s+ | 登录会明显变慢，不建议 |

\* 实际耗时取决于服务器 CPU，需在目标机器实测。**经验法则：单次哈希控制在 200–500ms**——对用户登录无感，但让离线暴力破解成本上升数个数量级。

### 1.4 实施要点（设计层面，不含代码改动）

1. **encode 显式指定 cost**：注册 / 改密 / 重置时，用 cost = 12 生成盐再哈希。位置：`PasswordUtil.encode()`。
2. **matches 无需改**：BCrypt 哈希串里已内嵌 cost 和盐，`BCrypt.checkpw` 校验时自动读取，**老的 cost=10 哈希仍可正常校验**，新写入的才是 cost=12。
3. **平滑升级（可选增强）**：现有逻辑已对「历史 SHA-256」做了登录后升级。可顺势扩展为「检测到哈希 cost 低于目标值时，也在登录成功后用新 cost 重新哈希」，让存量用户逐步迁移到 cost=12，无需强制改密。
4. **cost 设为可配置常量**：将目标 cost 提取为 `PasswordUtil` 内的常量（或读 `application.yml`），便于后续随硬件升级调高，无需改多处。
5. **登录接口压测**：cost 提高后单次登录 CPU 开销上升，需评估高并发登录场景（如批量脚本登录）的 CPU 占用，必要时配合第 2、3 节的限流与验证码降低无效请求量。

### 1.5 关于 Argon2id（备选，本次不采用）

Argon2id 是密码哈希竞赛冠军，抗 GPU/ASIC 能力强于 BCrypt（内存硬）。但 Hutool 不直接支持，需引入 BouncyCastle 等依赖并做迁移。**本次按 BCrypt 提高 cost 即可满足需求**，Argon2id 列为未来可选演进方向。

## 2. 接口防护：让 F12 看不到明文 + 防篡改防重放

目标拆成三层，从易到难叠加：

```
[A] 传输层加密  → F12 抓到的密码是密文，明文不出现在请求体
[B] 请求签名    → 请求体被篡改即拒绝（防中间人改参数）
[C] 防重放      → 抓包后原样重发无效（防重放攻击）
```

### 2.0 地基：必须先有 HTTPS

任何应用层加密都建立在 **HTTPS（TLS）** 之上。没有 HTTPS，攻击者在网络链路上就能截获一切，应用层再加密也只是抬高门槛。

- 生产环境强制 HTTPS，HTTP 请求 301 跳转 HTTPS。
- 响应头加 `Strict-Transport-Security`（HSTS），强制浏览器后续只走 HTTPS。

> 注意：HTTPS 之后，F12 的 Network 面板**仍能看到解密后的明文请求体**（浏览器在 TLS 解密之后展示）。所以要让 F12 里看不到明文密码，必须叠加 [A] 应用层加密。

### 2.1 [A] 传输层加密：密码不以明文进请求体

**方案：非对称加密传输密码字段。** 推荐 **RSA**（生态成熟，Hutool 直接支持）或 **SM2**（国密，合规场景）。

流程：

```
1. 前端启动时 GET /api/auth/public-key  → 拿到后端 RSA 公钥（公钥可公开，F12 看到也无所谓）
2. 用户输入密码 → 前端用公钥加密 password → 得到 Base64 密文
3. 登录请求体提交密文：{ username, password: "<密文>" }
4. 后端用私钥解密还原明文 → 再走 BCrypt.matches 校验
```

要点：

- **私钥只在后端**，绝不下发。公钥下发无风险。
- F12 Network 里 `password` 字段是一串密文，**看不到明文**——达成用户诉求。
- 公钥建议**定期轮换**（如随服务启动生成，存 Redis，设过期），密钥对生成用 Hutool `SecureUtil.generateKeyPair("RSA")`。
- 注册、改密接口同样套用。
- **重要澄清**：这不是「加密就无法破解」。前端 JS 里的加密逻辑 F12 可见，攻击者能照搬。它的价值是：① 明文密码不落在抓包/日志/浏览器历史里；② 配合 [B][C] 形成纵深防御。

> 命名提醒：当前 `RegisterDTO.passwordHash` 字段名有误导（实际传明文）。引入传输加密后，建议统一字段语义——要么真的传密文并改名为 `encryptedPassword`，要么保持 `password`。避免「名字叫 hash 实际是明文」的混淆。

### 2.2 [B] 请求签名：防篡改

**方案：HMAC 签名。** 客户端对「请求参数 + 时间戳 + nonce」按约定规则拼接后做 HMAC-SHA256，签名随请求头发送，后端用同样规则重算比对。

签名串构造（示例约定）：

```
待签名串 = HTTP方法 + "\n" + 请求路径 + "\n" + 排序后的参数串 + "\n" + timestamp + "\n" + nonce
signature = HMAC_SHA256(待签名串, secretKey)
```

请求头携带：

```
X-Timestamp: 1730000000000      # 毫秒时间戳
X-Nonce: 8f3a...                 # 一次性随机串（UUID 或随机 16 字节）
X-Sign: <HMAC-SHA256 结果, Base64/hex>
```

后端校验：重算签名，与 `X-Sign` 比对，不一致即 `401/400` 拒绝。

- **作用**：请求体 / 参数被中间人篡改后签名对不上，直接拒绝。
- **secretKey 的现实约束**：纯前端（公开 Web）的 secret 必然可从 F12 读到，所以前端签名**只能防「无脑篡改/普通用户」，挡不住有心人逆向**。它真正的强约束场景是「服务端到服务端」「带签名的 OpenAPI」。对本项目，前端签名作为**纵深防御的一层**仍有价值（提高门槛、配合防重放），但不要把它当成不可破。
- secret 不要硬编码成明显常量，可结合登录态 token 派生，进一步抬高逆向成本。

### 2.3 [C] 防重放：抓包重发无效

仅有签名还不够——攻击者把**整个合法请求（含签名）**原样重发，签名依然有效。需要时间戳 + nonce 双重约束：

1. **时间戳窗口**：后端校验 `X-Timestamp` 与服务器时间差，超过阈值（如 ±5 分钟）拒绝。挡住「很久以后再重放」。
2. **nonce 去重**：把 `X-Nonce` 存入 **Redis，TTL = 时间戳窗口**（如 5 分钟）。
   - 请求进来先查 Redis 是否已存在该 nonce → 存在则判定重放，拒绝。
   - 不存在则写入 Redis（`SETNX` + 过期），放行。
   - TTL 与时间戳窗口一致即可：窗口外的请求已被时间戳拦掉，nonce 无需长期保留，不会撑爆 Redis。

项目已依赖 `spring-boot-starter-data-redis`，直接复用即可。

### 2.4 在哪里接入（贴合现状）

后端统一在**拦截器**里做签名 + 防重放校验，避免每个 Controller 重复写。现有拦截器注册在 `WebConfig.addInterceptors()`（已注册 `SaInterceptor` 和 `MaintenanceInterceptor`）：

```
新增 ApiSignInterceptor，注册到 /api/** （或仅敏感接口如 /api/auth/**）
执行顺序建议：维护拦截器 → 签名/防重放拦截器 → Sa-Token 鉴权拦截器
```

- 校验失败统一抛 `BusinessException`，由 `GlobalExceptionHandler` 转成标准 `ApiResult` 错误响应。
- 登录、注册接口本身不需要登录态（已在 `excludePathPatterns` 里），但**应纳入签名 + 防重放 + 验证码**保护，因为它们是暴力破解的主要目标。

### 2.5 这套方案能挡住什么 / 挡不住什么

| 威胁 | 是否缓解 | 说明 |
| --- | --- | --- |
| F12 直接看到明文密码 | ✅ 挡住 | [A] 传输加密后是密文 |
| 抓包看到明文密码 | ✅ 挡住 | HTTPS + [A] |
| 中间人篡改请求参数 | ✅ 挡住 | [B] 签名 |
| 抓包后原样重放 | ✅ 挡住 | [C] 时间戳 + nonce |
| 撞库 / 暴力破解登录 | ⚠️ 部分 | 需配合第 3 节验证码 + 登录限流 |
| 攻击者逆向前端 JS 仿造请求 | ❌ 挡不住 | 前端代码公开，只能抬高门槛 |
| 数据库被拖库 | ✅ 缓解 | 第 1 节 BCrypt 哈希 |

**一句话**：以上让「随手抓包/改包/重放」基本失效，把攻击者门槛从「打开 F12」抬到「逆向前端 + 实时计算签名」，但**根除是不可能的**，安全是提高成本而非绝对。

## 3. 自研滑块缺口验证码

目标：登录前先过一道「拖动滑块把拼图对到缺口」的人机验证，挡住自动化脚本撞库。**自研**（不引第三方库），后端用 Hutool 画图，Redis 存校验状态，前端 Vue 自绘组件。

### 3.1 整体流程

```
1. 前端打开登录页 / 点击验证框 → GET /api/captcha/slider
       ↓ 后端：随机背景图 + 随机缺口位置，挖出缺口、生成滑块小图
2. 后端返回：{ captchaId, bgImage(带缺口的背景,base64), sliderImage(滑块,base64), sliderY }
       ↓ 后端把「正确的缺口 X 坐标」存 Redis：key=captchaId, value=correctX, TTL=2分钟
3. 用户拖动滑块 → 前端记录滑动轨迹和最终 X 坐标
4. 前端 POST /api/captcha/verify { captchaId, x, trail }
       ↓ 后端：取 Redis correctX 比对 |x - correctX| <= 容差；校验轨迹；通过则发一次性 ticket
5. 后端返回 captchaTicket（一次性令牌，存 Redis，TTL 短）
6. 登录请求带上 captchaTicket → 后端校验 ticket 有效且未用过 → 删除 ticket → 放行登录
```

关键设计点：**缺口的正确坐标永远只在后端**，前端只拿到「带缺口的图」，肉眼可见但程序难自动定位（需图像识别）。校验结果通过**一次性 ticket** 串到登录接口，避免验证码和登录脱节。

### 3.2 后端设计

**接口一：获取验证码** `GET /api/captcha/slider`

- 准备一组背景底图（放 `resources` 下，或纯色/噪声生成）。
- 随机生成缺口位置 `(correctX, sliderY)`，`correctX` 在 [滑块宽, 图宽-滑块宽] 范围内。
- 用 Hutool `Img` / Java `BufferedImage` 在背景图上：① 按缺口形状抠出滑块小图；② 在原图缺口处叠加半透明遮罩形成「凹槽」。
- 生成 `captchaId`（UUID）。
- **Redis 存**：`captcha:slider:{captchaId}` → `correctX`，TTL 2 分钟。
- 返回（坐标只回 Y，X 严禁返回）：

```json
{
  "captchaId": "uuid",
  "bgImage": "data:image/png;base64,...",
  "sliderImage": "data:image/png;base64,...",
  "sliderY": 45,
  "bgWidth": 310,
  "sliderWidth": 50
}
```

**接口二：校验滑动** `POST /api/captcha/verify`

```json
请求：{ "captchaId": "uuid", "x": 187, "trail": [[x,y,t], ...] }
```

校验步骤：

1. 从 Redis 取 `correctX`，**取不到**（过期/不存在）→ 失败。
2. **位置校验**：`|x - correctX| <= 容差`（建议 5~8 px）。
3. **轨迹校验（反机器人核心）**：
   - 机器人常是**匀速/瞬移**直达，人类有**加速—减速—微调**的非线性轨迹和轻微抖动。
   - 检查：是否存在加减速变化、是否有回拉/抖动、总耗时是否在合理区间（太快如 <300ms 可疑）、Y 轴是否有自然的微小波动。
   - 轨迹校验可由松到严分阶段上线，先上位置校验，再逐步加轨迹规则。
4. **一次性消费**：无论成功失败，**立即删除** `captcha:slider:{captchaId}`，防止同一验证码重复试错（暴力猜 X）。
5. 通过 → 生成 `captchaTicket`（UUID），存 Redis `captcha:ticket:{ticket}` → `username 或 1`，TTL 5 分钟，返回给前端。

**接口三（融合进登录）**：登录请求体增加 `captchaTicket`

- `AuthService.login` 前置校验：Redis 查 `captcha:ticket:{ticket}` 存在 → **校验后立即删除**（一次性）→ 继续登录；不存在 → 抛「请先完成滑块验证」。
- 这样验证码与登录强绑定，且 ticket 用完即焚，无法复用。

### 3.3 防作弊要点（自研容易踩的坑）

| 坑 | 后果 | 对策 |
| --- | --- | --- |
| 把 `correctX` 返回给前端 | 脚本直接读坐标秒过 | **X 只存 Redis，绝不返回** |
| 验证码可重复提交 | 脚本暴力猜 X（容差内总能撞中） | 校验后**立即删 Redis**，一次一用 |
| 只校验位置不看轨迹 | 脚本算出 X 直接提交即过 | 加**轨迹行为校验** |
| 验证码与登录脱节 | 过了验证码但登录可被单独刷 | 用**一次性 ticket** 串联，登录时消费 |
| 无过期时间 | Redis 堆积 + 验证码长期有效 | 所有 key 设 **TTL** |
| 缺口图过于简单 | 图像识别轻松定位缺口 | 背景加噪声/干扰、缺口边缘做混淆 |

> 诚实提示：自研滑块能挡住**绝大多数普通脚本**，但挡不住「接入打码平台 + 图像识别」的专业黑产。若未来对抗升级，再考虑商业风控或成熟开源方案（如 AJ-Captcha）。当前自研方案是**成本与防护的合理平衡**。

### 3.4 前端组件设计（Vue 3 + naive-ui）

- 新建 `SliderCaptcha.vue` 组件，登录按钮点击前弹出或内嵌在表单上方。
- 结构：背景图（带缺口）+ 可拖动滑块 + 底部滑轨。
- 交互：
  1. `onMounted` / 点击刷新 → 调 `/api/captcha/slider`，渲染 `bgImage`，滑块按 `sliderY` 定位。
  2. 监听滑块 `mousedown/touchstart` → `mousemove/touchmove` 记录 `trail`（每个点存 `[x, y, 相对起始的时间戳]`）→ `mouseup/touchend` 提交。
  3. 提交 `x`（滑块移动的最终位移）+ `trail` 到 `/api/captcha/verify`。
  4. 成功 → 拿到 `captchaTicket`，显示「验证通过」，把 ticket 存入登录表单状态。
  5. 失败 → 提示「验证失败，请重试」，**重新拉取新验证码**（旧的已被后端删除）。
- 移动端：同时绑定 `touch` 事件，支持触屏拖动。
- 无障碍：滑块加 `aria-label`，并提供「刷新验证码」按钮。

### 3.5 与登录流程的整合

```
登录页加载
  → 拉滑块验证码
用户拖动滑块通过
  → 拿到 captchaTicket，存入 loginForm
用户点「登录」
  → 前端：① 用公钥加密 password（§2.1）
          ② 生成签名头 X-Timestamp/X-Nonce/X-Sign（§2.2/2.3）
          ③ 请求体带 captchaTicket
  → 后端拦截器：验签 + 防重放（§2.4）
  → 后端 AuthService：消费 captchaTicket → 解密 password → BCrypt.matches（§1）
  → 登录成功，发 Sa-Token
```

三块能力在登录这一条链路上叠加，形成纵深防御。

## 4. 落地优先级建议

| 优先级 | 事项 | 工作量 | 价值 |
| --- | --- | --- | --- |
| P0 | BCrypt cost 提到 12 + 低 cost 平滑升级（§1） | 小 | 拖库防护立竿见影 |
| P0 | 生产强制 HTTPS + HSTS（§2.0） | 小（运维） | 一切加密的地基 |
| P1 | 自研滑块验证码（§3） | 中 | 挡住脚本撞库 |
| P1 | 登录密码传输加密 RSA（§2.1） | 中 | F12 看不到明文 |
| P2 | 请求签名 + 防重放拦截器（§2.2/2.3/2.4） | 中大 | 防篡改防重放，纵深防御 |

建议按 P0 → P1 → P2 推进，每步独立可上线、独立可验证。

## 5. 写在最后的安全观

- **没有绝对安全**：前端代码永远可被 F12 读到，所有客户端逻辑都可被逆向。安全工程的目标是**持续提高攻击成本**，让攻击的代价远高于收益。
- **纵深防御**：单一手段都有破绽（前端签名可逆向、滑块可被打码平台破），但多层叠加显著抬高综合门槛。
- **服务端是最后防线**：任何客户端校验都只是「减少无效流量」，真正的安全判断（密码校验、权限、限流）必须在服务端完成且不可被客户端绕过。
