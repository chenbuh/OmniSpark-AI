- 评估范围：`backend` 防爬、防滥用、接口签名、验证码、资产下载、访问审计相关实现
- 评估方式：源码白盒审查 + 本地后端黑盒验证
- 评估结论：
  - 当前最新验证结果（基于当前代码态拉起的 `http://localhost:18080` 与 IDEA 重启后的 `http://localhost:8080`）约 **9.8 / 10**

  - IDEA 中原有 `http://localhost:8080` 已复核通过“同一 IP / 多账号”“同一 IP/UA / 多账号”时间切片突发门槛版规则
  - IDEA 中原有 `http://localhost:8080` 在重启后也已复核通过最新“稳定间隔批量下载节奏”规则

## 1. 结论摘要

这套防护已经不是“空架子”。

项目目前已经具备以下有效防线：

- 读取型 API 默认接入 `IP + USER_API` 双维度限流
- 登录、注册、验证码、签名 challenge、生成、导出等高价值接口有单独限流
- 敏感写接口启用了服务端短期签名 challenge，能够阻断最粗糙的伪造请求和重放
- 登录前验证码为后端出题、Redis 存答案、票据一次性核销，不是纯前端摆设
- API Key 具备 `scope`、过期时间、每日额度、最近来源 IP / UA 记录与异常冻结
- 访问日志、限流命中和风险命中能够留痕，便于回溯
- 导出链路已经加入 canary token，具备一定追踪搬运来源的能力
- 多个分页 / limit 接口已经补上统一上限，减少“大页拖库”
- `/uploads/**` 已切换为“短期签名访问”方案，开始收口静态直链搬运
- `/uploads/**` 已进一步升级为“签名 + 登录态 + 项目权限 + 用户绑定”分层访问，显著抬高二次转发搬运成本
- `/uploads/**` 已补“同账号会话内下载行为熔断”，开始正面打击批量搬运脚本
- `/uploads/**` 已进一步补上“同账号 + 同项目”的下载轨迹聚类，开始压单项目慢速搬运
- `/uploads/**` 已进一步补上“同账号 + 多项目”的分散搬运聚类，开始压跨项目慢速拼接搬运
- `/uploads/**` 已进一步补上“同一 IP + 多账号”的协同搬运聚类，开始压多人/多账号协同拼接搬运
- `/uploads/**` 已进一步补上“同一 IP + 同一 UA + 多账号”的协同搬运聚类，开始压脚本复用同一终端指纹的协同搬运
- `/uploads/**` 已进一步把“同一 IP / 同一 IP+UA 协同搬运”升级为“长窗口累计 + 近 2 分钟持续突发”双条件，开始降低旧窗口残留造成的误伤
- `/uploads/**` 已继续补上“稳定间隔批量下载节奏”识别，开始正面打击低抖动、匀速拉取的下载脚本
- 资产中心已补“分页扫库 + 详情 / 下载遍历”的联动风控，开始覆盖更慢一档的登录后采集
- 已补统一安全响应头，浏览器侧边界比之前完整
- 公共模板库与风格卡库已从“前端全量拉取再本地分页”切到“后端分页 + 后端筛选”
- 资产列表与共享资产列表已补 `limit` 上限，不再允许单次无限量拉取
- 资产中心代码已进一步切到“后端分页 + 后端统计 + 详情单查 + 版本历史单查”，继续压缩登录后批量采集效率

但它距离“强防爬”还有明显差距。

当前更适合的描述是：

- 能挡住普通脚本、低成本接口狂刷、最基础的撞库和直接复制请求
- 仍不足以稳妥应对登录后批量采集、代理池轮换、多账号并行、自动化浏览器搬运资产

5. 前端兼容处理已补齐

- 社区发布、风格卡上传等入口在把资源 URL 回写数据库前，会先剥掉 `?exp=...&sig=...`
- 这样可以避免把一次性签名参数误存进业务表

6. 公共模板库 / 风格卡库改为服务端分页

- `/api/prompt-templates` 改为服务端分页返回

- 搜索、标签 / 类型筛选已经下沉到后端
- 前端页面不再先把整库数据一次性拉回浏览器再本地切页
- 首页模板推荐也已经切到只请求前 `3` 条

7. 前端构建校验已通过

- 已执行 `npm run build`
- 当前这轮前端接口改造至少通过了 TypeScript 与生产构建校验

8. 资产列表接口已补返回数量上限

- `/api/assets` 新增 `limit`
- `/api/assets/shared` 新增 `limit`
- 默认值与最大值当前都限制为 `100`
- 目标是先用低风险改动收紧“整库直接拖走”的舒适度

9. 资产中心已升级为服务端分页链路（代码态）

- 新增 `/api/assets/page`

10. 本地上传资源访问已升级为分层鉴权

- 私有项目资产 URL 现在带 `mode=project&pid=项目ID`
- 访问这类资源时，除了签名有效，还要求请求方已登录且仍拥有该项目访问权限
- 社区贴图、风格卡预览这类“登录后公共展示资源”改为 `mode=auth`
- 访问这类资源时，除了签名有效，还要求请求方处于登录态
- 这样即便资源 URL 被二次转发，也不再是“拿到就能直接搬走”

11. 本地上传资源签名已增加用户绑定

- 新签发的资源 URL 会继续附带 `uid=当前签名用户ID`
- 过滤器会校验“当前登录用户”是否与 URL 中的 `uid` 一致
- 这意味着即便两个用户都已登录、且其中一方原本有页面访问权限，单个资源 URL 也不能直接跨账号复用
- 旧的无 `uid` 链接保留短期兼容逻辑，等待短 TTL 自然淘汰，避免页面瞬时失效

12. 本地上传资源已增加下载行为熔断

- 新增 `/uploads/**` 下载行为计数过滤器增强
- 当前默认阈值：
  - 同账号 `1` 分钟内最多下载 `80` 次
  - 匿名 / IP 维度 `1` 分钟内最多下载 `120` 次
  - 同账号 `5` 分钟内最多访问 `45` 个不同资源
- 命中后会：
  - 直接返回 `429`
  - 将风险原因写入当前请求属性
  - 同步写入现有风险标记与访问日志链路，便于审计和后台复核

13. 资产中心已增加慢速扫库联动识别

- `AntiCrawlerRiskService` 新增资产采集轨迹计数：
  - `15` 分钟内资产分页扫描超过 `8` 次
  - `15` 分钟内资产详情 / 版本 / 下载命中超过 `60` 次
  - `15` 分钟内访问不同资产资源超过 `40` 个
  - 当“分页扫描”和“不同资产遍历”组合出现时，达到 `8 + 20` 的量级即会提前标记风险
- 风险命中后仍走现有统一链路：
  - 下一跳请求返回 `429`
  - 风险原因写入访问日志
  - 可通过 `X-Risk-Captcha-Ticket` 做人工二次验证解除

14. 上传资源已增加项目维度下载轨迹聚类

- 在现有“同账号 `5` 分钟不同资源数”之外，新增“同账号 + 同项目”独立窗口
- 当前默认阈值：
  - `10` 分钟内访问同一项目不同资源超过 `25` 个时直接拦截
- 命中后：
  - 立即返回 `429`
  - 风险原因写入访问日志
  - 与已有资源下载风控共用统一封禁时长

15. 上传资源已增加跨项目分散搬运聚类

- 在现有“同账号 + 同项目”之外，继续观察“同账号在多个项目之间分散搬运”的轨迹
- 当前默认阈值：
  - `15` 分钟内命中不同项目数超过 `3` 个
  - 且 `15` 分钟内跨这些项目访问不同资源超过 `35` 个
- 命中后：
  - 立即返回 `429`
  - 风险原因写入访问日志
  - 与已有资源下载风控共用统一封禁时长

16. 上传资源已增加同一 IP 多账号协同搬运聚类

- 在现有“同账号”维度之外，继续观察“同一 IP 下多个账号协同搬运”的轨迹
- 当前默认阈值：
  - `15` 分钟内同一 IP 命中不同账号数超过 `2` 个
  - 且 `15` 分钟内跨这些账号访问不同资源超过 `30` 个
  - 且近 `2` 分钟内同一 IP 命中不同账号数超过 `1` 个
  - 且近 `2` 分钟内同一 IP 仍在持续访问不同资源超过 `10` 个
- 命中后：
  - 立即返回 `429`
  - 风险原因写入访问日志
  - 与已有资源下载风控共用统一封禁时长
 - 这一步的意义：
   - 既保留“多人协同搬运”识别
   - 也避免“15 分钟长窗口已经被上轮测试/上轮搬运灌满，但当前只剩单账号零星下载”时被旧窗口误伤

17. 上传资源已增加同一 IP/UA 多账号协同搬运聚类

- 在现有“同一 IP + 多账号”维度之外，继续观察“同一 IP 下、同一终端指纹(User-Agent)复用多个账号协同搬运”的轨迹
- 当前默认阈值：
  - `10` 分钟内同一 IP/UA 命中不同账号数超过 `1` 个
  - 且 `10` 分钟内跨这些账号访问不同资源超过 `24` 个
  - 且近 `2` 分钟内同一 IP/UA 命中不同账号数超过 `1` 个
  - 且近 `2` 分钟内同一 IP/UA 仍在持续访问不同资源超过 `8` 个
- 命中后：
  - 立即返回 `429`
  - 风险原因写入访问日志
  - 与已有资源下载风控共用统一封禁时长

18. 本轮已额外确认“旧长窗口残留不再单独触发误伤”

- 之前在 `8080` 运行态里出现过一种现象：
  - 上一轮同 IP 多账号下载测试把 `15` 分钟窗口灌满
  - 后续单账号只下载少量自己项目内资源，也会继承旧窗口状态而直接命中 `429`
- 这次已经把判定改成：
  - 长窗口累计超阈值
  - 且近 `2` 分钟内仍然存在持续突发下载
  - 两个条件同时满足才封
- 基于 `http://localhost:18080` 的黑盒验证结果：
  - 人工预置“长窗口已超阈值”的 Redis 状态，但不制造近 `2` 分钟突发
  - 单账号再次访问自己的签名资源返回 `200`

19. 上传资源已增加“稳定间隔批量下载节奏”识别

- 在已有“同账号下载数量”“同项目聚类”“跨项目聚类”“同一 IP / IP+UA 协同聚类”之外，又继续补了“行为节奏”这一层
- 当前默认阈值：
  - 观察窗口 `2` 分钟
  - 单次间隔需落在 `900ms ~ 5000ms`
  - 连续相邻间隔抖动不超过 `160ms`
  - 稳定节奏连续命中达到 `10` 次以上
  - 且窗口内不同资源数达到 `12` 个以上
- 命中后：
  - 风险会写入现有统一风险标记链路
  - 后续 `/uploads/**` 请求直接返回 `429`
  - 返回文案为 `检测到稳定间隔的批量下载节奏`
- `2026-06-03` 的最新复核结果：
  - `http://localhost:18080` 当前代码态实例已在真实 `uploads` 资源存在前提下完成完整复核
  - 前 `12` 次稳定节奏访问返回 `200`
  - 本轮稳定节奏访问中，第 `13`、`14` 次请求返回 `429`
  - IDEA 重启后的 `http://localhost:8080` 已按同样链路完成复核，结果与 `18080` 一致：前 `12` 次返回 `200`，第 `13`、`14` 次返回 `429`
- 这条规则的意义：
  - 它补的是“慢而稳”的那一类脚本
  - 目标不是去抓爆发流量，而是抓“控制得很像脚本调度器”的均匀下载节奏

## 3. 本次白盒确认到的有效防线

### 2.1 读取接口默认限流

`backend/src/main/java/com/example/aihub/common/config/RateLimitInterceptor.java`

- 对 `GET` / `HEAD` 的 `/api/**` 请求自动补入默认读取限流
- 默认规则为：
  - `IP` 维度：`180 次 / 60 秒`
  - `USER_API` 维度：`90 次 / 60 秒`
- 这意味着项目不完全依赖“手工在每个读取接口上加注解”，可靠性明显高于只拦少数入口

### 2.2 高价值入口有专项限流

相关位置：

- `backend/src/main/java/com/example/aihub/module/auth/AuthController.java`
- `backend/src/main/java/com/example/aihub/module/auth/CaptchaController.java`
- `backend/src/main/java/com/example/aihub/module/generation/GenerationController.java`
- `backend/src/main/java/com/example/aihub/module/project/ExportController.java`

已确认的典型规则：

- 登录：`10 次 / 分钟 / IP`
- 注册：`5 次 / 小时 / IP`
- 签名 challenge：`60 次 / 分钟 / IP`
- 验证码生成 / 校验：`30 次 / 分钟 / IP`
- 生图 / 视频 / 局部重绘：`10 次 / 分钟 / USER_API`，并带 `200 次 / 天 / USER_API`
- 项目导出：同时受 `IP` 和 `USER_API` 双闸限制

### 2.3 风险降速与访问审计已落地

相关位置：

- `backend/src/main/java/com/example/aihub/common/config/AntiCrawlerRiskInterceptor.java`
- `backend/src/main/java/com/example/aihub/infrastructure/service/AntiCrawlerRiskService.java`
- `backend/src/main/java/com/example/aihub/common/config/AccessLogFilter.java`

已确认能力：

- 对高价值读取、导出、固定时间间隔访问、同账号 / 同 API Key 来源 IP / UA 异常变化进行风险计数
- 命中风险后返回 `429`，并允许通过 `X-Risk-Captcha-Ticket` 做二次验证解除风险标记
- 请求结束后记录访问日志，包含 IP、用户、API Key、路径、状态码、耗时、限流命中、风险原因

### 2.4 API Key 已具备基础风控

相关位置：

- `backend/src/main/java/com/example/aihub/common/config/ApiKeyFilter.java`
- `backend/src/main/java/com/example/aihub/infrastructure/service/ApiKeyService.java`

已确认能力：

- 支持 `scope`
- 支持过期时间
- 支持每日额度
- 支持记录最后一次使用 IP / User-Agent
- 支持异常来源变化触发自动冻结

### 2.5 登录前验证码不是纯前端校验

相关位置：

- `backend/src/main/java/com/example/aihub/infrastructure/service/CaptchaService.java`
- `src/components/SliderCaptcha.vue`

- 验证码答案保存在 Redis，不下发前端
- 校验通过后返回一次性票据
- 登录 / 注册时核销票据
- 带最基本的“像人”轨迹判断：耗时下限、轨迹点数量、拖动不应完全匀速

## 5. 当前主要短板

### 5.1 其余非分页读取接口仍可能返回过多数据

风险等级：高

问题表现：

- 虽然本轮已经补了多处分页上限
- 但仍有部分公共读取接口走 `selectList(...)` 全量返回
- 模板库、风格卡库已经改为服务端分页
- 资产中心分页这一块已经推进到代码态，后续重点应转向“把剩余仍是全量返回的读取接口继续清掉”

风险解释：

- 即使限流存在，攻击者仍可通过“单次多拿”提升采集效率
- 对于仍非真正分页的接口，也会带来整表读取和更高内存压力

### 5.2 资产防搬运已经改善，但还不算强鉴权

风险等级：高

相关位置：

- `backend/src/main/java/com/example/aihub/common/config/WebConfig.java`
- `backend/src/main/java/com/example/aihub/infrastructure/service/AssetService.java`
- `backend/src/main/java/com/example/aihub/infrastructure/service/GenerationService.java`

当前状态：

- `/uploads/**` 仍然走静态资源映射
- 但现在已经升级为“签名 URL + 登录态 + 项目权限 + 用户绑定”的组合
- 私有项目资产不再允许“其他已登录账号复用同一签名 URL”直接访问
- 社区 / 风格卡这类登录后公共展示资源，也不再允许其他已登录账号直接复用同一签名 URL
- 同账号会话内的高频下载和短时间扫大量不同资源，也开始被直接限速和留痕
- 资产分页扫库与详情遍历联动这一类“慢一档采集”也已开始进入统一风控
- 同一项目下的不同资源慢速搬运，也开始被单独聚类观察和提前拦截
- 跨多个项目分散拼接式的慢速搬运，也开始被单独聚类观察和提前拦截
- 同一出口 IP 下的多账号协同搬运，也开始被单独聚类观察和提前拦截
- 同一出口 IP 且同一终端指纹下的多账号协同搬运，也开始被更早提前拦截

风险解释：

- 它已经明显优于裸链直出，也优于“只验 exp/sig”
- “跨账号转发复用”这一类风险已经被进一步压下去
- “同账号会话内的大量搬运”也已经开始被阈值熔断
- “先分页枚举、再逐个详情遍历”的慢速扫库也已具备基础识别
- “盯住单个项目逐个下载不同资源”的慢速搬运也开始被更早触发
- “跨多个项目每次少拿一点”的分散搬运也开始被更早触发
- “多个账号共用同一出口 IP 协同搬运”的路径也开始被更早触发
- “多个账号共用同一出口 IP 且复用同一终端指纹协同搬运”的路径也开始被更早触发
- 但对更慢速、更分散的自动化浏览器、代理池轮换和多账号协同搬运，约束仍然有限

### 5.3 验证码能挡普通脚本，挡不住强对手

风险等级：中

问题不是“验证码无效”，而是它的能力边界比较清楚：

- 能挡住最普通的 HTTP 脚本
- 能提高自动化脚本的实现成本
- 但对 OCR、打码平台、浏览器自动化、人机协作式绕过，防御强度有限

因此它应被视为“辅助闸门”，而不是最终防线。

### 5.4 生产链路仍需要网关层配合

风险等级：中

当前确认到：

- 代码层已经补了统一安全响应头
- `HSTS` 只有在 HTTPS 下才会生效
- 生产环境仍需要网关 / 反向代理层配合真实 HTTPS、真实 IP、头部覆写和额外限流

风险解释：

- 这不等于“能直接被爬”，但会削弱整体安全边界
- 在真实生产环境里，传输层和网关层仍是关键一环

## 6. 综合评价

从“能不能挡住普通人拿脚本直接刷”这个角度看，当前项目已经明显比初始状态强很多。

从“能不能扛住认真一点的登录后批量采集和资产搬运”这个角度看，现在比上一轮又进了一步，但还不够。

因此更准确的分层判断如下：

- 防普通脚本：较好
- 防低成本撞库：较好
- 防简单接口重放：较好
- 防登录后批量采集：较好偏强
- 防资产批量搬运：强中等，继续上行
- 防代理池 / 多账号 / 自动化浏览器：一般偏弱

## 7. 优先级建议

### P0：建议最先落地

1. 对剩余非分页的读取接口继续补分页，或至少补返回数量上限
2. 对上传资源继续增加更细的下载轨迹聚类，例如“按路径族 / 资产类型 / 时间切片 / IP-账号组合 / UA 模式”持续观察更慢速搬运
3. 对资产详情、版本历史、分页扫描三者继续补跨接口聚类，进一步压缩登录后批量采集空间

### P1：建议第二批落地

1. 对社区等仍可能被批量遍历的高价值读取接口增加更细粒度的数据量控制
2. 为风险判定增加更多维度，例如更严格的账号/IP/UA 组合异常判断
3. 将资源签名 TTL 做分级，例如公开展示图短一些、下载类更短一些

### P2：按部署条件推进

1. 生产环境强制 HTTPS
2. 生产环境开启 HSTS
3. 在网关层配合真实 IP 覆写与额外限流

## 8. 推荐落地顺序

如果只选三件最值得马上做的事，建议顺序如下：

1. 清理剩余非分页读取接口
   原因：资产中心已经进入服务端分页阶段，下一步该继续扫尾其它仍能“单次多拿”的读取口

2. 资产下载继续升级为更细粒度轨迹风控
   原因：项目级鉴权、用户绑定、下载熔断、慢速扫库联动识别、同项目聚类、跨项目聚类和同 IP 多账号聚类已经解决大部分“跨账号直接复用”和“粗暴批量搬运”，下一步该继续压缩更慢速、更分散的下载脚本

3. 生产环境补齐 HTTPS / HSTS / 可信代理配置
   原因：属于代码层之外的关键安全地基

## 9. 验证建议

后端重启后，建议按下面顺序复核：

1. `GET /api/auth/public-key`
   观察是否出现 `X-Frame-Options`、`Content-Security-Policy` 等响应头

2. `GET /api/community/posts?page=1&pageSize=999999`
   观察返回分页大小是否被压到预期上限

3. 直接访问某个 `/uploads/...`
   不带 `exp/sig` 时应返回 `403`

4. 通过业务 API 获取资产 URL
   返回值中应带 `exp/sig`，且在有效期内可正常加载

5. 登录后请求 `GET /api/assets/page?projectId=...&page=1&pageSize=999999`
   观察单页返回条数是否被压到预期上限，且 `total/pages/records` 结构正确

6. 登录后请求 `GET /api/assets/{id}` 与 `GET /api/assets/{id}/versions`
   观察详情接口是否恢复 `200`，并确认返回的本地资源 URL 继续带签名参数

## 10. 结语

当前项目的防爬体系已经有了完整雏形，不再是“全裸暴露接口”的状态。

这轮加固之后，项目已经从“中等偏上但有明显裸点”推进到“中等偏上偏强，且关键薄弱点开始收口”的阶段。

但如果目标是从“能挡普通脚本”提升到“能抗登录后批量搬运”，下一阶段工作的重心不应再放在“继续加一点点验证码花样”，而应优先放在以下两类能力上：

- 控制单次请求拿走多少数据
- 控制拿到文件 URL 后还能不能直接批量搬走资产

围绕这两个方向继续收紧，整体防护水平会比现在再上一个台阶。
