# 密码加密设计方案

> 本文档现同时作为**设计方案 + 实施记录**。立论依据为 **2026-05-10** 快科技援引的卡巴斯基密码暴力破解研究报告，结合本项目现状给出算法选型、落地方案与当前实施状态。
>
> 项目现状：后端 Spring Boot 3 + Sa-Token + MyBatis-Plus + Redis + Hutool；密码哈希集中在 `PasswordUtil`，当前已用 BCrypt（兼容历史无盐 SHA-256，登录成功后平滑升级）；登录接口 `/api/auth/login`，注册 `/api/auth/register`。

## 0. 当前实施状态（2026-05-31）

已落地：

- `PasswordUtil.encode()` 已显式使用 `BCrypt.gensalt(12)`，新密码统一按 cost=12 存储。
- 登录成功后会自动升级两类旧密码：历史无盐 SHA-256，以及 cost 低于 12 的 BCrypt 哈希。
- 注册、个人改密、管理员创建用户、管理员重置密码、CSV 导入用户，已统一接入同一套密码策略校验。
- 密码最小长度已提升到 `>= 8`，并拦截常见弱密码及“与账号相同”的密码。
- 前端注册页、设置页、管理员创建用户页的提示文案和校验规则已同步更新。
- 注册请求字段已统一为 `encryptedPassword`；旧字段 `passwordHash` 已移除。
- 已新增 `/api/auth/public-key` 公钥下发接口；登录、注册、个人改密、管理员手动设初始密码已改为前端 RSA-OAEP 加密后提交。
- 明文降级通道已关闭：后端默认 `app.security.password.require-encrypted=true`，仅接受密文字段 `encryptedPassword` / `encryptedOldPassword` / `encryptedNewPassword`，收到明文密码字段直接拒绝；需排障或紧急回退时可临时置为 `false`。
- `/api/auth/**` 与管理员用户写操作已补上 `X-Timestamp + X-Nonce + X-Sign` 请求签名校验，并通过 Redis 做 nonce 去重防重放；nonce 去重 TTL 已设为 `2×窗口+1s`，覆盖时间戳 `[now-窗口, now+窗口]` 整个可接受跨度，消除窗口边界的秒级重放竞态。
- 对 `application/x-www-form-urlencoded` 敏感请求，前后端已统一改为“按参数名排序后的规范串”参与签名，避免表单请求在框架包装后出现验签不一致，个人改密链路已完成实测。

待落地：

- 更细粒度的签名密钥轮换 / 登录态派生 secret（当前仍为前后端约定 secret，且前端 bundle 内可见，主要作用是抬高篡改与重放门槛，挡不住有心伪造）。
- **多实例部署前置项**：RSA 密钥对当前在服务启动时随进程生成，单实例可用；负载均衡多实例下，A 实例下发的公钥与 B 实例的私钥不匹配会导致解密失败。多实例前需改为共享密钥对（配置注入或从 Redis / 配置中心加载固定密钥）。
- 生产强制 HTTPS + HSTS：否则中间人可替换 `/api/auth/public-key` 下发的公钥，架空传输加密。

## 1. 背景：为什么必须重视密码哈希算法

卡巴斯基 2026-05 发布研究报告，用**一块英伟达 RTX 5090 显卡**对超过 **2.31 亿个** 以 **MD5** 算法哈希存储的唯一密码做暴力破解测试，结果：

| 指标 | 数据 |
| --- | --- |
| 一小时内被攻破 | **60%** |
| 60 秒内被攻破 | **48%** |
| 含义 | 每 10 个 MD5 保护的账户，拿到哈希后 6 个会在 1 小时内沦陷 |

佐证趋势：

- **2024 年 Hive Systems**：RTX 4090 约 **17 秒** 即可破解一个「大小写字母 + 数字」组成的 8 位密码。
- **卡巴斯基去年测试**：单张 RTX 4090 一小时内可破解 **59%** 的哈希密码。
- **攻击成本极低**：攻击者无需自购 RTX 5090，可通过云服务**按小时租用**同规格 GPU 发起攻击。

**根因**：MD5（以及 SHA-1/SHA-256 等）是**为计算速度而设计**的通用摘要算法，GPU 可并行以**每秒数十亿次**的速率比对哈希、还原明文。一旦密码数据库泄露（拖库），这类哈希形同虚设。

**卡巴斯基的建议**：服务商全面弃用 MD5，改用 **bcrypt、Argon2** 这类**专为密码存储设计的「慢速」算法**；用户侧坚持「一账号一密码，越长越随机」。

> 本项目当前已使用 BCrypt（见 `PasswordUtil`），方向正确。本方案要做的是：**确认并强化** BCrypt 配置，清除历史弱哈希残留，并补齐用户侧密码策略。

## 2. 威胁模型：本方案防的是什么

| 攻击场景 | 密码哈希能否缓解 | 说明 |
| --- | --- | --- |
| **拖库后离线暴破**（本文重点） | ✅ 核心防护 | 慢哈希 + 加盐让 GPU 暴破成本上升数个数量级 |
| **彩虹表查表** | ✅ 加盐即失效 | 每个用户独立随机盐，预计算表无效 |
| 撞库（用别处泄露的明文登录） | ⚠️ 哈希无关 | 需登录限流 + 验证码（见 `安全加固设计方案.md`） |
| 抓包看到明文密码 | ⚠️ 哈希无关 | 需 HTTPS + 传输加密（见第 6 节） |

**一句话**：密码哈希的核心价值是「**即使数据库被拖走，攻击者也难以还原出明文密码**」。这正是卡巴斯基报告打击的场景。

## 3. 算法选型：弃用 MD5/SHA，选慢哈希

### 3.1 三类算法对比

| 算法 | 类型 | 抗 GPU 暴破 | 加盐 | 本项目可用性 | 结论 |
| --- | --- | --- | --- | --- | --- |
| MD5 / SHA-1 / SHA-256 | 快速摘要 | ❌ 极弱（每秒数十亿次） | 需手动 | Hutool 支持 | **禁用于密码** |
| **BCrypt** | 慢哈希 | ✅ 强（故意慢 + 不规则内存访问） | ✅ 自带 128bit 随机盐 | Hutool `cn.hutool.crypto.digest.BCrypt` 原生支持 | **推荐，本项目已采用** |
| Argon2id | 慢哈希（内存硬） | ✅ 最强（抗 GPU/ASIC） | ✅ 自带盐 | Hutool 不直接支持，需引 BouncyCastle | 未来演进可选 |
| scrypt | 慢哈希（内存硬） | ✅ 强 | ✅ 自带盐 | 需额外依赖 | 备选 |

### 3.2 为什么选 BCrypt（而非 Argon2）

- **卡巴斯基明确推荐** bcrypt 与 Argon2 二选一，二者都能有效抵御暴破。
- BCrypt 在本项目**零新增依赖**：Hutool 已内置，且 `PasswordUtil` 已在用，改动最小、风险最低。
- BCrypt 哈希串**自带版本、cost、盐**（格式 `$2a$<cost>$<22位盐><31位哈希>`），校验时自解析，天然支持「不同 cost 共存、平滑升级」。
- Argon2id 抗 GPU 更强（内存硬），但需引入 BouncyCastle 并做迁移，**本次不采用**，列为未来硬件对抗升级时的演进方向。

### 3.3 BCrypt 为什么能抗住 RTX 5090

- **故意慢**：基于 Blowfish 密钥扩展，迭代轮数由 cost 控制，单次哈希耗时是 MD5 的**百万倍量级**，GPU 的「每秒数十亿次」优势被直接抹平。
- **内存访问不规则**：对 GPU 大规模并行不友好，无法像 MD5 那样流水线化。
- **自带随机盐**：彩虹表失效，每个用户哈希唯一，无法「破一个通杀一片」。

## 4. BCrypt 落地设计

### 4.1 cost 因子（work factor）

cost 以 2 的幂控制迭代轮数，**每 +1 耗时翻倍**。Hutool 的 `BCrypt.hashpw(raw)` 默认 cost = 10，偏低。

| cost | 相对耗时 | 单次哈希参考* | 评价 |
| --- | --- | --- | --- |
| 10 | 1x | ~50–100ms | Hutool 默认，偏低 |
| **12** | 4x | ~200–400ms | **推荐**，安全与体验平衡 |
| 13 | 8x | ~400–800ms | 高安全场景 |
| 14 | 16x | ~1s+ | 登录明显变慢，不建议 |

\* 实测耗时取决于服务器 CPU，需在目标机器实测。**经验法则：单次哈希控制在 200–500ms**——用户登录无感，但离线暴破成本上升数个数量级，直接击穿卡巴斯基报告里的「1 小时破 60%」前提。

### 4.2 实施要点（设计层面）

1. **encode 显式指定 cost**：注册 / 改密 / 重置时用 `BCrypt.hashpw(raw, BCrypt.gensalt(12))`，而非默认的 `hashpw(raw)`。位置：`PasswordUtil.encode()`。
2. **cost 提为可配置常量**：在 `PasswordUtil` 定义 `BCRYPT_COST = 12`（或读 `application.yml`），便于随硬件升级调高，避免散落各处。
3. **matches 无需改**：BCrypt 哈希串内嵌 cost 和盐，`BCrypt.checkpw` 自动解析，**老的 cost=10 哈希仍能正常校验**，新写入的才是 cost=12，新旧无缝共存。
4. **彻底清除 MD5/SHA 残留**：确认全项目无任何地方用 MD5/SHA 存储或比对密码（历史 `PasswordUtil` 曾有无盐 SHA-256，现已改造，需复查无遗漏）。

### 4.3 存量用户平滑升级（无需强制改密）

现有逻辑已对「历史 SHA-256」做了登录后升级（`AuthService.login` 中 `isLegacyHash` 判断后重写为 BCrypt）。建议**扩展该机制**，覆盖「低 cost BCrypt」：

```
登录校验通过后：
  if 哈希是无盐SHA-256（历史遗留）          → 用 cost=12 重新哈希并落库
  else if 哈希是BCrypt 且 cost < 目标值(12)  → 用 cost=12 重新哈希并落库
  else                                       → 不动
```

- 用户**无感知**，在下次成功登录时自动迁移到 cost=12。
- 无需运维批量重置密码，也不会锁死老用户。
- 解析已有哈希的 cost：BCrypt 串第 2 段即是（`$2a$10$...` 中的 `10`）。

## 5. 用户侧密码策略（呼应卡巴斯基建议）

卡巴斯基强调用户侧「一账号一密码、越长越随机」。服务端可通过策略引导：

| 策略 | 设计 | 说明 |
| --- | --- | --- |
| **最小长度** | 从当前 5 位提到 **≥ 8 位**（建议 10+） | 8 位字母+数字在 RTX 4090 上约 17 秒破，长度是抗暴破第一要素 |
| **复杂度** | 建议（非强制）含大小写 + 数字 + 符号 | 强制过严会逼用户用规律密码，反而更弱 |
| **弱密码黑名单** | 拒绝 `123456`、`password`、与用户名相同等常见弱口令 | 撞库/字典攻击首选这些 |
| **强度提示** | 注册/改密时前端实时显示强度条 | 引导而非强制，体验更好 |
| **泄露库比对（可选增强）** | 对接 HaveIBeenPwned k-匿名接口校验密码是否已泄露 | 防「一密码撞开所有账户」 |

> 该项已落地：注册/改密已统一要求密码 ≥ 8 位，并加常见弱密码黑名单与“不得与账号相同”的校验。这是改动小、收益高的一步。

## 6. 配套：密码传输加密（防 F12 / 抓包看明文）

> 密码哈希解决的是「**拖库后**」的安全，但密码在**传输途中**仍可能以明文出现在 F12 Network、代理日志里。两者需配套，不可互相替代。

1. **HTTPS 是地基**：生产强制 HTTPS + HSTS，否则链路上一切明文可截。
2. **应用层加密密码字段**：前端先调用 `/api/auth/public-key` 获取 **RSA 公钥**，再将密码加密后以 `encryptedPassword`（改密时为 `encryptedOldPassword`、`encryptedNewPassword`）提交，后端用私钥解密还原明文，再走 `BCrypt.matches`。F12 里看到的是密文字段，不再直接出现明文密码。
   - 私钥仅在后端，公钥可公开。
   - 当前实现为服务启动时生成 2048 位 RSA 密钥对，并通过 Web Crypto `RSA-OAEP + SHA-256` 在浏览器侧加密。
3. **诚实边界**：前端 JS 加密逻辑 F12 可见、可被逆向，传输加密只能保证「明文不落在请求体/日志」，**无法根除**客户端逆向。它的价值是抬高门槛 + 配合签名防重放。

> 此节与 `docs/安全加固设计方案.md` 第 2 节内容衔接，详细的请求签名 / 防重放设计见该文档，避免重复。

| --- | --- | --- | --- |

## 8. 小结

- 卡巴斯基用 RTX 5090 一小时破 60% 的 MD5 密码，证明**通用快速摘要（MD5/SHA）绝不能用于密码存储**。
- 解法是**慢哈希**：本项目沿用 **BCrypt** 并把 **cost 提到 12**，零新增依赖、改动最小，即可让 GPU 暴破成本上升数个数量级。
- 配套清除历史弱哈希、平滑升级存量用户、收紧密码长度策略、加上传输层加密，形成「**存储 + 传输 + 用户策略**」的完整密码安全闭环。
- 没有绝对安全：哈希防拖库、传输加密防抓包、长随机密码防撞库，多层叠加才是正解。安全工程的目标是**让攻击成本远高于收益**。
